### 为什么线上导入用不了/加载缓慢

[电子书PDF文件来自Github知名项目Chinatextbook。](https://github.com/TapXWorld/ChinaTextbook/ "点击跳转") 因此“线上导入”功能会访问Github服务器。

如果你可以连接到Github服务器，但是下载时间巨长无比（还表现为“处理中”窗口动画卡顿），

那么你可以在“设置”页面中将“Github下载加速”改为[ghfast](https://ghfast.top "ghfast")。

如果你无法连接到github服务器，


**简而言之，请使用VPN。**

　

###为什么启用音频拉伸后音频播放速度不变？

首先，您需要安装[FFMPEG](https://ffmpeg.org/ "ffmpeg官网"), 

其次，您需要先开启“音频拉伸”，然后再点击生成音频。

　

###为什么我修改了xx设置，但没有生效？

请点击“应用设置”。

注：屎山堆得有点多，暂未实现“自定义字体（及字号）”功能。

# 程序特色介绍
## 从DOCX文件导入”功能
使用该功能时，用户可以在弹出的窗口中选择DOCX文档并导入，

而 __无需打开其他任何软件__ 。



 
今年八月，开发团队随机采访了一些使用过同类软件的用户（九成为学生，一成为文职员工），

发现其中有很多用户未使用过带有此功能的同类免费软件。

(注：为了“圈钱"，很多软件已向VIP开发了此功能，但VIP性价比很低。）

团队还发现约四成同类软件用户

由于使用了不含微软Office的第三方“精简”系统而只能使用冷启动时间较长的金山WPS，因此白白浪费了时间。

（注：这四成用户在采访前都不知道如何在不给流氓软件交“智商税"的情况下手动安装微软Office）

因此我们开发了此功能，为用户省时省心。


 

该功能实现并不复杂，但 __成功将传统的__
```
找到需要导入的文件→双击打开专用编辑器（微软OfficeWord/金山WPS等）→（等待软件加载）→全选并复制→回到文本转语音软件→粘贴

```
（__依赖专用编辑器__ 且不考虑“寻找文件”与“移动鼠标”过程的话，全过程约需要 __十五秒__ （WPS实测）
若用户电脑老旧则 __会在第二步等待更长时间__）
这 __六步__ 操作简化为
```
找到需要导入的文件→单击选择→点击确定
```
__三步__ 操作（不考虑寻找文件与移动鼠标过程的话全过程最多耗费 __两秒__），
__大幅减少了用户操作软件所需时间。__

　

## 从图片导入功能
使用该功能时，
用户可以在弹出的窗口中选择图片，
程序会自动发送给AI进行文字识别并获取结果。

　

随着移动设备普及，用户获取信息的场景逐渐“视觉化”：

__纸质文件__ 拍照存档、

__屏幕截图__ 保存内容（如 __网课笔记截图__ 、网页片段）、

__手写笔记__ 拍照记录等。

这些图片中

往往包含需要转换为可编辑文本的信息，

但 __传统“手动输入”方式效率极低__
例如一张包含1000字连续文本的图片，
普通人手动录入平均 __需15分钟以上，__
且 __易因看错、漏输导致错误__（注：数据来源于AI）。
即使是精通盲打的 __专业文员 也需要约8分钟__ 才能完成录入
(注：数据来源就是上一部分提到的，采访时遇到的文职员工)

　

同时，市场上独立的OCR（光学字符识别）工具普遍存在两类问题：

__操作复杂__：需单独下载工具、上传图片、等待识别、手动复制结果，再切换到语音合成工具粘贴，多环节切换增加用户操作成本；

__识别局限性__：对来自照片或手写文字的公式符号、序号标记识别准确率低需用户手动校对，反而增加工作量。

比如来讲：<img width="1549" height="624" alt="image" src="https://github.com/user-attachments/assets/91a81869-a995-44dc-940a-c912eab5ee30" />

原始内容是：(1+𝑥)^𝑛=1+𝑛𝑥/1!+(𝑛(𝑛−1) 𝑥^2)/2!+…，

输出是：n(n-)x2
		(1-x)n=1+++=

原始内容与输出内容大相径庭，
严重影响用户正常使用。

　

即使是用上了豆包等AI大语言模型软件，
也存在着步骤繁琐的问题 (注：相较于本功能)

　

不难发现，
大部分用户需要一个 __”一站式"__ 解决方案：
在语音合成工具内
##### 直接完成“图片→文字→语音”的全流程，同时将识别错误率降低到无限接近于0。
减少工具切换与二次编辑成本，
##### 避免因识别错误导致的不可预料的损失。


　



## __ 在线获取教科书内容并导入 __ ##
使用该功能时，用户可以一站式



　

#### 获取几乎全部中国教科书PDF资源，并指定页码和所需的内容。



作为学生，
项目组自该程序开发之初
就将学生群体作为主要目标用户群体。

　


在开发过程中，我们思考了这样一个问题：
对于一名学生来讲，
有什么问题是在学习生活中简单易懂，但又枯燥乏味的呢？

同时，作为一名开发者，什么问题是我们能够轻易解决的呢？

　

我们的答案是：
#### 把“抄写”作业变成“听写”。
对，就这么简单。

　

此项功能将传统的
```
拿出手机→找出教科书→翻到对应的页面→用手机给对应页面拍照→将照片传入电脑→使用正确率低的传统OCR识别文字→复制结果→粘贴进输入框
```
（  __依赖手机__ 且不考虑“翻书、找到手机并打开相机app、搜索并打开OCR网站”与“移动鼠标”过程的话，
所有需要用户手动进行的操作 __共需要耗费1-2分钟__ ）
这 __八步操作__ 

　

简化为
```
点击“在线导入"→选择对应的教科书pdf文件→输入需要的信息
```
__三步操作__
（ __不依赖手机__ 且 __全过程最短仅需15s__)

　

该功能有效利用抄写时”听觉”这一空闲的感官，

 ### 节省了用户抬头看书所需时间的同时，辅助用户更快的记忆书本上的知识，帮助用户更高效地学习。
